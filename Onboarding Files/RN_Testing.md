# Writing Unit and Integration Tests for React Native

Testing is important in React Native development because it ensures that your app works correctly before it reaches real users. Mobile apps are used by many different people on many different devices, and a bug that slips through can seriously damage the user experience and the reputation of the app. In a real project like Focus Bear, where users depend on the app to manage their productivity and daily habits, a broken feature can be very frustrating. Testing catches these bugs early in the development process when they are much cheaper and easier to fix. Testing also gives developers confidence when making changes or adding new features because if something breaks, the tests will immediately tell you exactly what went wrong and where. Without tests, every change you make requires you to manually check the entire app to make sure nothing broke, which is slow and unreliable.

Mocking API calls in tests means replacing the real network request with a fake one that returns data you control. In Jest, you do this using jest.mock("axios") at the top of your test file, which replaces the entire axios library with a fake version. Then you use axios.get.mockResolvedValue() to tell the fake axios exactly what data to return when it is called. This way, your tests never actually connect to the internet or depend on a real server being online. For example, in our UserProfile tests, instead of making a real call to jsonplaceholder.typicode.com, we told the fake axios to return a fake user object with a name, email, and phone number. We also tested what happens when the API fails by using axios.get.mockRejectedValue(), which tells the fake axios to pretend the request failed with an error. This allowed us to test both the success and failure states of our component without needing a real server at all.

Unit tests and integration tests serve different purposes and test different levels of your application. A unit test checks one single piece of code in complete isolation, meaning it does not depend on anything else working correctly. For example, in our project, the math utility tests are unit tests because they only test one function at a time, like checking if add(2, 3) returns 5, without involving any other part of the app. Integration tests, on the other hand, check how multiple parts of your app work together. For example, our UserProfile test is an integration test because it tests how the component works together with the mocked API, the state management, and the rendering, all at the same time. The key difference is that unit tests are simpler, faster, and easier to debug because there is only one thing that can go wrong, while integration tests are more realistic because they test how everything connects the way a real user would experience it. In a well-tested project, you need both unit tests to make sure each piece works correctly and integration tests to make sure all the pieces work correctly together.
