#  Handling API Calls in React Native using Axios & Axios-Retry

Axios is preferred over fetch in many cases because it makes API calls simpler, safer, and more powerful with less code. One of the biggest advantages of Axios is that it automatically throws an error when the server responds with a bad status code like 404 or 500. With fetch, you have to manually check response.ok and throw the error yourself, which means it is very easy to forget and accidentally treat a failed request as a success. Axios also automatically converts the response data to JSON for you, while fetch requires you to call response.json() as a separate step. Another major advantage is that Axios supports interceptors, which allow you to run code before every request is sent or after every response is received. This is very useful for things like automatically adding an authentication token to every request or logging errors in one central place.

Axios-Retry improves network reliability by automatically retrying failed requests instead of immediately giving up and showing an error to the user. In real mobile apps, network failures happen all the time — the user might be on a weak WiFi connection, switching between networks, or the server might be temporarily overloaded. Without retry logic, a single failed request means the user sees an error even if the request would have succeeded if tried again a second later. Axios-Retry solves this by intercepting failed requests and automatically sending them again up to a specified number of times. You can configure how many times it retries, how long it waits between each attempt, and which types of errors should trigger a retry. For example, in our axiosInstance.js, we configured it to retry up to 3 times with an increasing delay of 1 second, 2 seconds, and 3 seconds between attempts. We also configured it to only retry on network errors or server errors, not on client errors like 404, since retrying a "not found" request would never succeed anyway. This makes the app feel much more reliable to users because temporary network hiccups are handled automatically without the user ever knowing something went wrong.

Handling API failures gracefully in a React Native app means making sure the user always sees a helpful message and never gets stuck on a broken screen when something goes wrong. The first step is to always wrap API calls in a try/catch block so that errors are caught and handled instead of crashing the app. Inside the catch block, you should update the component state to show a friendly error message to the user instead of showing a blank screen or a technical error. For example, in our PostList.js component, we set an error state when the API call fails and display the message "Failed to load posts. Please try again," which is much better than showing nothing. Another important practice is to always show a loading state while the API call is in progress so the user knows something is happening and does not think the app is frozen. You should also use response interceptors in Axios to handle common errors in one central place — for example, if the server returns a 401 unauthorized error, you can automatically redirect the user to the login screen from the interceptor instead of handling it in every single component.
